## ปัญหาและวิธีแก้

### 1. ปุ่ม AP ไม่ส่ง field `action` ไป backend
- **อาการ:** กด "บันทึกข้อมูล" (SD) ข้อมูลบันทึกถูกต้อง แต่กด "ยืนยันส่งขั้นตอนถัดไป" (AP) ไม่มี field `action` ใน request เลย
- **Root cause:** `onSave()` เดิม build `params` inline เฉพาะ branch SD (ส่งเข้า `onSaveDraft`) ส่วน branch AP เรียก `Sentflow()` ที่ build `submission` ของตัวเองแยกก้อนกันโดยสิ้นเชิง — ไม่มี `TASK_CODE`/`TASK_APPROVE`/`backendId`/`action` เลย เป็น **duplicated payload-building logic** ตรงกับ pattern ที่เจอใน form-104 เมื่อ 2026-07-10 ([[Duplicated Payload-Building Logic Drifts Out of Sync]])
- **Fix:** ดึง logic build payload ออกเป็น private method `_buildParams(action: string)` ตัวเดียว ให้ทั้ง `onSaveDraft` (SD) และ `Sentflow` (AP) เรียกใช้ร่วมกัน พร้อมแนบ `action: action` เข้าไปทุก path — เป็นการ implement จริงตาม "ทางแก้ระยะยาว" ที่เคยแนะนำไว้ในโน้ตเดียวกันตั้งแต่ 2026-07-10 (ตอนนั้นยังเป็นแค่คำแนะนำ)
- Commit: `fix(flow3-task131): แก้ AP flow ไม่บันทึก action + คำนวณวันขยายเวลาใหม่` (`15fcec2e`)

### 2. กด AP แล้วไม่เข้า breakpoint ของ backend (Visual Studio 2022)
- **อาการ:** ตั้ง breakpoint ใน backend controller ไว้ กดปุ่ม AP breakpoint ไม่ติด ทั้งที่กด SD ติดปกติ
- **วิธี diagnose:** เปิด DevTools Network tab เช็คก่อนว่า request ออกจริงไปที่ไหน — พบว่า POST ไปที่ `localhost:5000/complete` และได้ response 200 กลับมาจริง (หลังปรับ debug flag ใน `environment.staging.ts` ให้ชี้ไป localhost) แปลว่า request "ถึง" backend แล้วจริง ปัญหาไม่ใช่ routing/host แต่เป็นเรื่องอื่น (debugger attach ผิด process/build หรือ breakpoint อยู่คนละ endpoint)
- **สรุปสาเหตุที่แท้จริง:** SD กับ AP ยิงไปคนละ backend endpoint กัน — SD → `Updateform` ของ `FormInspectionSummaryService`, AP → endpoint ที่ resolve แบบ dynamic จาก `formConfig.apiSubmit` ผ่าน `FlowService.sentFlow()` — ตรงกับ pattern ที่เคยเจอมาก่อนใน form-10-flow4 เมื่อ 2026-07-07 ([[Service Endpoint Mismatch Blocks Field From Persisting]]): FE ส่ง field ถูกต้อง แต่ endpoint ปลายทางคนละตัวไม่ persist field เดียวกันลง DB
- **ยังไม่ปิดเคส:** ต้องเช็คโค้ด backend (.NET, นอก repo frontend) ว่า endpoint `complete`/`apiSubmit` ของ Task131/135/139 map field `action` เข้า column ของ table `base` แล้วหรือยัง
- สรุปวิธี diagnose เป็น checklist ทั่วไปไว้ที่ [[Debugging Backend Breakpoint Not Hit Checklist]]

### 3. `onExtensionDayChanged` แปลงจาก arrow-function class property เป็น regular method
- **สังเกตในโค้ด:** เดิม `onExtensionDayChanged = (e: any) => {...}` (auto-bind `this`) ถูกเปลี่ยนเป็น `onExtensionDayChanged(e: any) {...}` (regular method) และฝั่ง template ต้องเพิ่ม `.bind(this)` ตอนอ้างอิงใน `editorOptions.onValueChanged` เพื่อชดเชย
- **หลักการที่ยืนยันตรงกับโน้ตเดิม:** [[DevExtreme Callback this Undefined]] — DevExtreme widget เรียก callback โดยไม่ผูก `this` ให้อัตโนมัติ ถ้าไม่ใช้ arrow-function class property ต้อง `.bind(this)` เองเสมอที่จุดอ้างอิง ไม่งั้น `this` ข้างในจะกลายเป็น undefined/widget instance แทน component — เป็นทางเลือกที่ถูกต้องเทียบเท่ากัน แค่ต้องจำ bind เอง

### 4. Refactor logic คำนวณรอบขยายเวลา (`EXTENSION_ROUND`/`EXTENSION_DAY`/`EXTENSION_AMOUNT_DAY`)
- เปลี่ยนชื่อ field เดิม `EXTENSION_NEW_DAY`/`EXTENSION_COUNT`/`EXTENSION_NEW_DUE_DAY` เป็นชื่อใหม่ให้ตรงกับ backend และเปลี่ยนแหล่งข้อมูลจาก `Get()` เป็น `GetPhinai5History()` เฉพาะ Task131/135/139 เพื่อคำนวณรอบขยายเวลาจาก history ก่อนหน้า (`IS_CLOSED` = SD คือยัง draft ใช้รอบเดิม, = AP คือปิดรอบแล้วต้อง +1 รอบถัดไป)
- เป็น business logic เฉพาะโปรเจกต์ ไม่ extract เป็น pattern ทั่วไป (เก็บไว้แค่ในโน้ตนี้)
