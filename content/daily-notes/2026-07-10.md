### 1. crime-scene-location-redesign: map และ lat/lng ไม่แสดงตอนย้อนกลับมาหน้าเดิม
- **อาการ:** กด "ระบุพิกัด" แล้วปักหมุด → กด "ถัดไป" ไปหน้าอื่น → กด "ย้อนกลับ" มาหน้าเดิม แผนที่กับ field ละติจูด/ลองจิจูดหายไป ทั้งที่ `formData` ยังมีค่าพิกัดอยู่
- **Root cause 3 ชั้น:**
  1. getter `showCoordinatePicker` มี side-effect `this._isMapVisible = !this._isMapVisible` อยู่ข้างใน — getter ถูก Angular เรียกทุก change detection cycle (ไม่ใช่แค่ตอนคลิก) ทำให้ state สุ่มพลิกกลับไปมาโดยไม่ตั้งใจ
  2. field lat/lng ใช้ `*ngIf="_isMapVisible"` ซึ่งเป็น local state ที่ reset เป็น `false` ทุกครั้งที่ component ถูกสร้างใหม่ (ตอนย้อนกลับ) ทั้งที่ `formData` (จาก parent) ยังมีพิกัดอยู่
  3. parent เรียก `mapComponent.showMap()` ซ้ำจากทั้ง `ngOnChanges` และ `ngAfterViewInit` — ในขณะที่ child (`leaflet-map-overview`) ก็เรียก `this.showMap()` เองใน `ngAfterViewInit` อยู่แล้ว เพราะ `showMap()` เป็น toggle (`isMapVisible = !isMapVisible`) การถูกเรียกซ้ำทำให้แผนที่ถูกสร้างแล้วลบทิ้งทันที
- **Fix:**
  - ลบ side-effect ออกจาก getter ให้เป็น pure getter
  - เพิ่ม logic ใน `ngOnChanges`: ถ้า `formData` มี lat/lng อยู่แล้ว ให้ set `_isMapVisible = true`
  - ลบการเรียก `mapComponent.showMap()` ซ้ำใน `ngOnChanges`/`ngAfterViewInit` ของ parent เหลือแค่ child จัดการ init เอง + ปุ่ม "ระบุพิกัด" เรียก toggle แบบตั้งใจ
- Commit: `fix: แก้ map และ lat/lng ไม่แสดงเมื่อย้อนกลับมาหน้าข้อมูลที่เกิดเหตุ` (dc35e2bf)
- **หลักคิดสำคัญ:** getter ห้ามมี side-effect (violate การคาดเดาได้ของ Angular change detection) และ auto-init logic ควรอิงจาก "แหล่งความจริง" ที่ persist ข้าม component lifecycle (`formData`) ไม่ใช่ local field ที่ reset ทุกครั้ง

### 3. form-104: `Sentflow()` ส่ง `INVESTIGATION_START_TIME`/`COMPLETION_TIME` เป็น null ตอน "ยืนยันส่งขั้นตอนถัดไป"
- **อาการ:** กด "บันทึกข้อมูล" (SD) ค่าเวลาถูกบันทึกถูกต้อง แต่กด "ยืนยันส่งขั้นตอนถัดไป" (AP) ค่ากลายเป็น null
- **Root cause:** ปุ่ม AP เรียก `onSaveDraft(params)` ก่อน (params ถูก format เวลาแล้วถูกต้อง) แล้วค่อยเรียก `Sentflow()` แยกต่างหาก — แต่ `Sentflow()` build `params` ของตัวเองด้วย `{ ...this._formData, ...this._formDataio, ... }` โดย spread `_formDataio` (ข้อมูลจาก formio widget แยกต่างหาก ไม่มี field เวลาพวกนี้) ทับทีหลัง `_formData` ทำให้ค่าเวลาถูกเขียนทับเป็น `null`/`undefined` และไม่ผ่านการ format ด้วย
- **Fix:** เพิ่ม logic ใน `Sentflow()` ให้เขียนทับค่ากลับจาก `this._formData` (พร้อม `formatTimeToHHmm`) หลังจาก build `params` เสร็จ
- Commit: `fix: แก้ INVESTIGATION_START_TIME/COMPLETION_TIME เป็น null ตอนยืนยันส่งขั้นตอนถัดไป` (12d39bfb)
- **หลักคิดสำคัญ:** เมื่อ build request payload จากหลายแหล่งข้อมูล (`_formData` + `_formDataio`) ต้องระวังลำดับ spread — field ที่ authoritative source อยู่คนละ object กับที่ spread ทีหลัง จะถูกเขียนทับโดยไม่ตั้งใจ

### 4. form-104: `formatTimeToHHmm` คืนค่าว่างเมื่อ input เป็น string `"HH:mm"` ที่ format แล้ว
- **อาการ:** ต่อเนื่องจากข้อ 3 — หลัง fix ข้อ 3 แล้ว ยัง repro ได้กรณี "ไม่กดบันทึกรอบแรก กดยืนยันเลย" ค่าเวลายังเป็น null
- **วิธี debug:** เพิ่ม `console.log` 3 จุด (`onSave:entry`, `onSave:afterFormat`, `Sentflow:beforeSend`) เพื่อดูค่า/type ของ field เวลาในแต่ละขั้น ให้ user รันแล้ว copy log กลับมา
- **Root cause (เจอจาก log จริง):** `_formData.INVESTIGATION_START_TIME` ตอนเข้า `onSave()` เป็น **string `"16:11"` อยู่แล้ว** (โหลดมาจาก backend ตรง ๆ ไม่ใช่ Date object จาก dxDateBox เพราะ user ไม่เคยแตะ time picker) แต่ `formatTimeToHHmm()` ทำ `new Date(time)` กับทุก input โดยไม่เช็คก่อน — `new Date("16:11")` ใน JavaScript เป็น **Invalid Date** (ไม่ใช่ ISO date-time string ที่ valid) → `isNaN(d.getTime())` true → return `''`
- **Fix:** เพิ่ม guard ใน `formatTimeToHHmm` (shared helper) เช็คก่อนว่าถ้า input เป็น string ที่ตรง pattern `^\d{2}:\d{2}$` อยู่แล้ว ให้ return ค่าเดิมทันที ไม่ต้องผ่าน `new Date()`
- Commit: `fix: แก้ formatTimeToHHmm คืนค่าว่างเมื่อรับ string HH:mm ที่ format แล้ว` (b7581064)
- **หลักคิดสำคัญ:** function ที่ระบุ type union กว้าง (`Date | string | number`) ต้องเช็ค "รูปแบบที่ valid อยู่แล้ว" ก่อนพยายาม parse ซ้ำ — `new Date(validButNonISOString)` ไม่ throw error แต่คืน `Invalid Date` แบบเงียบ ๆ
